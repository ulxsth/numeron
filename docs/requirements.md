# Numer0n — 要件（暫定）

## 全体方針

- **下位レイヤー**: 「秘密の桁列」と「コール1回 → Hit/Blow を数える」だけの純粋なロジック。UI・対戦形式に依存させない。
- **上位レイヤー**: ターン制・履歴・双方向の秘密番号・**マッチ（BO 制）**・**アイテム**・開示ルールなどを載せる。

---

## 技術スタック

| 項目 | 内容 |
|------|------|
| 言語 | TypeScript |
| フロント | 第二弾でのデザインを想定し TypeScript + React.js を用いる。状態管理に Jotai、UI には Framer Motion を取り入れる。 |
| バックエンド | 自前サーバは置かず、**Supabase** でクライアント間の状態共有・同期を担う想定（DB、認証の要否、Realtime などは設計時に確定）。 |
| 分離 | Hit & Blow の純ロジックは Supabase や DOM に依存させず、単体テスト可能なモジュールとして切り出す。 |

---

## 第1段: Hit & Blow

### 目的

- 共有する「採点エンジン」と、**2人が別クライアントから参加できる最小ゲーム**まで。
- **Supabase で状態を共有し、Hit & Blow がオンラインで問題なく完走すること**を第1段の完了条件とする。

### ルール・パラメータ

| 項目 | 内容 |
|------|------|
| 桁・数字 | 桁数 3 または 4。各桁は 0–9。 |
| 重複 | 「桁内で重複なし」を基本とする。 |
| 先頭の 0 | 許す。 |
| 採点 | **Hit**: 位置も数字も一致。**Blow**: 数字は含まれるが位置が違う。重複なしなら `(Hit, Blow)` は一意。 |
| 入力検証 | 桁数・使用可能な数字・重複ポリシー違反はエラーとする。フォールバックで「正しく動いている」と誤認させない。 |

### 最小プレイ

- **双方がそれぞれ秘密を 1 つ持つ**。交互に相手に対するコールを行う。
- コールに対する Hit/Blow は **相手の秘密**に対する結果。
- **先に相手の番号を当てたプレイヤーが勝ち**（自分のコールで `Hit == N`）。
- ターン上限は任意（設定で付与可能）。

### 成果物イメージ

- `score(secret, guess) -> { hit, blow }` のような採点 API。
- 秘密の生成・入力の検証。
- ルーム（セッション）の生成・参加、**手番・双方向の秘密・コール履歴**の同期など、**Supabase を使ったオンライン用の薄いゲーム層**。
- 番組の **イート** 等は第2段で載せる（第1段は双方向・交互コールまで）。

## 第2段: Numer0n・演出

### 第2段の位置づけ

- 第1段でオンライン Hit & Blow（**双方向秘密・交互コール**）の土台が固まったあと、**Numer0n の番組寄せ**として **アイテム** と **BO 制マッチ**（下記）を実装し、**イート表示**や**画面・演出のデザイン**は並行または後続で本格化する。
- **接続の不安定さ・Realtime 通知の取りこぼし**に対する方針は、第1段の検証結果を踏まえてここで整理する（「非機能・スコープ」内の **信頼性** 参照）。

### ゲーム: Numer0n（Hit & Blow の上）

#### 目的

「相手の番号を当てる」双方向の典型形。テレビの細部ルールは後から差し替え可能にする。

#### 基本ルール

| 項目 | 内容 |
|------|------|
| 参加者 | 2 プレイヤー。第1段で確立したオンライン参加の仕組みを流用する。 |
| 秘密 | 各プレイヤーが自分の N 桁を1つ持つ。桁・重複・先頭0のポリシーは第1段と同一設定に揃える。 |
| ターン | 交互に「相手に対するコール」を1つ出す。 |
| 公開情報 | コールに対し、**相手の秘密**を基準に Hit/Blow を返す。 |
| 勝ち条件（単局） | 相手の番号を当てたプレイヤーがそのゲームの勝ち（自分のコールで `Hit == N`）。 |
| 勝ち条件（マッチ） | **BO 制**を有効にした場合、**先に W 本勝ったプレイヤーがマッチ勝ち**（例: Bo3＝2 本先取、Bo5＝3 本先取）。詳細は下節。 |
| ターン上限 | 単局ごとに任意（設定で付与可能）。 |

### 対戦形式（BO 制）

1 局だけだと、解の探索やアイテム運用の**メタが収束**しやすく単調になりがちなので、**マッチ単位で BO（ベスト・オブ）制**を選べるようにする。

| 項目 | 内容 |
|------|------|
| 用語 | **ゲーム（単局）**＝従来どおり、双方向の秘密とコールで 1 試合が始まり終わる単位。**マッチ**＝複数ゲームの勝敗を数え、全体の勝者を決める単位。 |
| 典型パラメータ | **先取 W 本**（例: Bo3＝2 本先取、Bo5＝3 本先取）。必要なら `2W-1` 局を上限として扱うなど、設計で明示する。 |
| Bo1 | **BO 無効**または W=1 とみなし、**現状の第1段に近い「1 セッション＝1 ゲーム」**としても動かせること。 |
| ゲーム間のリセット | 新しいゲームが始まるたびに、**秘密・コール／推測履歴・手番などは初期化**する。**アイテムカードの使用済みは BO マッチが終わるまで引き継ぐ**（各 `item_kind` はマッチ通算 1 回まで。`room_item_cards` を参照）。 |
| 先攻・手番 | マッチ内でゲームが進むときの先攻決め（毎ゲーム交互、毎ゲームランダム、マッチ固定など）は **1 通りに決めて** `supabase-design.md` 等に書く。 |
| 正本 | マッチスコア・現在のゲーム番号・マッチ終了条件は **DB で保持**し、クライアント表示と一致させる。 |

番組のトーナメントやロストナンバーによる跨ぎは本スコープでは必須としないが、**「数試合を通じて勝敗を付ける」体験**はこの層で担保する。

### アイテム（第2段の主実装スコープ）

番組に倣った**コール以外の行動**をオンラインで再現する。**今回の開発ではここを中心に要件化・実装する**想定。用語と番組との対応のたたき台は `docs/research/diff-hb-numer0n.md` を参照。

#### 方針

- **正本は DB**: 使用可否・効果の内容はクライアント任せにせず、Supabase 側（RLS・トリガー・RPC 等）で検証する。第1段のコールと同様、**フォールバックで成功を装わない**（失敗はエラー表示）。
- **ルール参照版**: 放送回によって HIGH&LOW 等の定義が異なる。**プロジェクトで採用する版（例: Wikipedia 項に概説されている現行形に近いルール）を1つ決め、設計・テストはその版に合わせる。**版を変えるときは本ドキュメントを更新する。
- **カード・ロストナンバー**: 番組の「数字カード在庫」「敗戦で数字を失う」は本スコープでは必須にしない。**アイテムカード**（6 種それぞれマッチ中 1 枚・使用で消費）は `room_item_cards` で表現する。秘密の生成は第1段と同様（0–9・重複なし・桁数固定）でよい。CHANGE の差し替え規則は別途定義する。
- **考慮時間・10秒ルール・会話での嘘**: テレビの進行用ルールは**必須にしない**。必要になった段階で追記する。

#### 対象にするアイテム

以下はいずれも番組に相当する役割を持つ。実装順・分期は設計で切ってよいが、**仕様の空白を残さないよう、採用版ごとに効果をこの要件に追記または別紙で固定する**。

| 区分（参考） | 名前 | ざっくりした効果（詳細は採用版で確定） |
|--------------|------|----------------------------------------|
| 攻撃 | DOUBLE | 自分のターンにコールを連続 2 回。代償として自分の秘密のうち相手が指定した桁を開示する。2 回分を 1 ターンとみなし、中間で相手のアイテム宣言が割り込まない形を番組に寄せる。 |
| 攻撃 | HIGH & LOW | 相手の各桁が HIGH（5–9）か LOW（0–4）かを知る。**採用版では「左から順に各桁の H/L を得る」形など、1 通りに固定する。** |
| 攻撃 | TARGET | 0–9 のうち 1 桁を指定し、相手番号に含まれるかを知る。含まれる場合は**桁位置まで**開示する（採用版で文言・例外を明示）。 |
| 攻撃 | SLASH | 相手番号の「最大桁の数字 − 最小桁の数字」および、採用版で定めるパターン情報を得る。 |
| 防御 | SHUFFLE | 自分の秘密の**桁の並び**だけを入れ替える（数字の集合は不変）。相手から見た Hit/Blow の推移が変わりうる。 |
| 防御 | CHANGE | 自分の秘密の 1 桁を、ルール上許される別の数字に差し替える。番組の HIGH/LOW 制約・カード前提を本スコープで採用しない場合は、**在庫なしの差し替え規則**を要件または `supabase-design.md` 側に書き切る。 |

#### ターンとアイテムの競合

- 番組では**攻撃系・防御系**に分類し、同一ターン内の組み合わせに制限がある。**初期実装**では「1 ターンにつきアイテムは 1 つまで」「DOUBLE 中は追加アイテム不可」など、**単純な制約から始めてよい**が、最終的に番組寄せで詰めたい挙動を本節か設計書に残す。
- コール・採点・勝敗判定との**順序**（アイテム宣言→効果適用→コール→手番交代など）はステートマシンとして文書化し、DB の更新順と一致させる。

#### 永続化・可観測性

- アイテムの使用履歴と開示結果は、**クライアントが捏造できない形**で保存する（専用テーブルまたは汎用イベントログ）。
- Realtime で対戦相手の UI が追従する前提。取りこぼしは「非機能・スコープ」の信頼性の節に従う。

#### 番組寄せのその他（アイテム以外・後続でも可）

- **イート（表示）**: Hit した桁の確定表示・以降の固定など。アイテムとは独立に設計できるが、**同一コールの禁止**などはコール／アイテム共通のルールとして整理してよい。
- **コール履歴**: コールと `(Hit, Blow)`、およびアイテム結果を区別して表示できることが望ましい。

---

## 非機能・スコープ

- **第1段**に **Supabase によるオンライン対戦** を含め、2クライアントで Hit & Blow が意図どおり終局まで動くことを確認する。
- AI 相手・ランキングは第1段では必須にしない。
- 採点などのコアロジックは、単体テストやローカル実行で Supabase なしでも検証できる分離を維持する。
- **Numer0n 固有の本格ルール（アイテム等）と演出・ビジュアル**は第2段で要件化し実装する（第1段は見た目より動作・整合性優先でよい）。

### 信頼性（再接続・通知の取りこぼし）

- **第1段では要件として深追いしない**。Realtime の届かなさ／切断復帰まわりは、クライアントのデフォルト挙動と **Postgres を状態の根拠にする**程度の前提で実装し、**動かして検証した事実**（再現条件、実際に困る境目）をメモしておく。
- **第2段で**、その観測に基づき、例として **再接続後の DB 再取得**、**ハートビート／`worker`／可視性変化への対応**、**Broadcast だけに寄せない**などの**指針を文書化・実装**する。第1段の段階で設計を固定しすぎない。

## 実装順（参考）

1. Hit & Blow コア（採点・秘密生成・バリデーションをテスト可能に分離）。
2. 手元での最小ループ（必要ならローカル2ペイン等。双方向・交互の挙動をコアで再現できていることの確認）。
3. **Supabase でセッション共有し、オンライン上で双方向 Hit & Blow が問題なく完走することを確認**（第1段の完了目安）。あわせて接続・通知まわりの事実観測を残す。
4. 信頼性の指針確定と実装（第1段の観測に基づく。必要なら DB 再取得・再接続時のUX 等）。
5. **アイテム**および **BO 制マッチ**（本ドキュメント「第2段」: DB 正・ターン制約・6 種の効果・履歴・マッチスコア）。スキーマ・RLS は `supabase-design.md` 等で追補する。
6. Numer0n のその他の番組寄せ（イート表示の本格化、同一コール禁止の細部など）。
7. 演出・デザイン、音声など体験面の仕上げ。
